01 Core

High-Level Platform

Build a Python FastAPI platform on Cloud Run. Every request starts with Entra-authenticated identity, becomes a scoped Quatta request, passes through AI Gateway policy, retrieves only permitted context and returns a grounded response with audit metadata.

Build

  • FastAPI API service.
  • AI Gateway service boundary.
  • PostgreSQL schemas for tenants, roles, source config, audit, cache and workflow state.
  • OpenTelemetry tracing across request, connector, retrieval and model calls.

Stack

  • Python, FastAPI, SQLAlchemy.
  • Google Cloud Run, Cloud SQL for PostgreSQL.
  • Vertex AI Gemini as default model provider.
  • Microsoft Entra ID for identity.

Acceptance

  • No connector call without auth context.
  • Every answer can be reconstructed from audit.
  • Missing or stale context is shown to the user.
  • Provider routing can switch to OpenAI/Azure OpenAI if needed.
02 Data access

Connector Layer

Connectors are read-only adapters. They fetch operational evidence from ERP, WMS, SQL, Excel/CSV, SharePoint, APIs and documents, normalize it and emit audit events. They do not make Quatta the new system of record.

SourceDefault methodKey control
ERP / WMSRead-only API, replica view or export table.Allowlisted endpoints, source timestamps and no write credentials.
SQLParameterized allowlisted queries.Tenant and role filters included in every query path.
Excel / CSVSharePoint, SFTP or upload ingestion.Column validation, freshness warning and source file version.
SharePoint / DocumentsMicrosoft Graph plus document ingestion.Document ACL, version, deletion and permission-change handling.

Contract

  • authenticate()
  • fetch(params)
  • normalize(records)
  • health_check()
  • emit_audit_event()

MCP Use

  • Use MCP for approved agent-callable tools.
  • Keep connector logic reusable outside MCP.
  • Expose clear schemas and read-only policies.

Acceptance

  • Every result has source ID, record ID and retrieval timestamp.
  • Failures are visible, never silently ignored.
  • Cache includes TTL and permission scope.
03 Context

Retrieval + RAG

Structured facts and document evidence use different retrieval paths. SQL/API retrieval remains structured for exact facts. Document evidence from SharePoint, API payloads, local uploads or SFTP drops is staged in tenant-scoped Google Cloud Storage before Document AI, chunking, embeddings, pgvector and metadata filters join facts in the Context Builder.

Structured Retrieval

  • Use for orders, stock, deliveries, tickets, claims and master data.
  • Return records with source, event time and retrieval time.
  • Use deterministic calculations for totals, liability and impact.

Document RAG

  • Stage raw documents in GCS with tenant IAM, KMS, lifecycle and quarantine status.
  • Use Document AI for PDFs, scans, contracts, PODs and SOPs.
  • Chunk by semantic sections and page ranges.
  • Create Vertex AI embeddings and store in pgvector.

Context Builder

  • Merge facts, passages, citations and warnings.
  • Include storage object references for document evidence.
  • Flag missing, stale or conflicting evidence.
  • Return a compact JSON context package for LangGraph.
04 Controls

Security, Governance and Audit

Security controls are cross-cutting. They are applied before and during data access, retrieval, model calls and human approval. They are not downstream dataflow steps.

Identity

  • Validate Entra ID tokens.
  • Map groups to Quatta roles.
  • Scope by tenant, site, customer, item, order, claim and document ACL.

Audit

  • Trace ID, user, role, source, row counts, document chunks, model, prompt version and policy result.
  • Denied checks and failed validations are logged.
  • Human approvals store reviewer and final version.
  • High-risk customers can export immutable audit evidence.

Fail Mode

  • High-risk flows fail closed if audit or permission checks fail.
  • Low-risk read-only flows may degrade only with explicit warning.
  • Cross-tenant retrieval must have automated tests.
  • KMS, DLP/classification, retention and data residency policies are tenant-aware.
05 Run

Deployment and Operations

Separate the runtime API, ingestion workers and outbound/export jobs. Give each service the minimum service account permissions it needs.

Services

  • Cloud Run API for app and gateway.
  • Cloud Run worker/job for document ingestion.
  • Cloud SQL PostgreSQL with pgvector.
  • Secret Manager for source credentials.

CI/CD

  • Build container images.
  • Run unit, connector contract and tenant-isolation tests.
  • Include IaC plan, security scan, migration checks and deploy approval.
  • Deploy with environment-specific config and service accounts.

Operations

  • Dashboards for latency, connector health, retrieval quality and model cost.
  • Alerts for ingestion backlog, failed connectors and audit failures.
  • Runbooks for revocation, stale data and provider outage.
  • Backup restore, SLO, quota and DR tests before scale.
06 Customer rollout

Customer Onboarding and Implementation Lifecycle

A Quatta rollout should start with source discovery and governance, not with an agent. Once the platform, permissions and retrieval are reliable, agents become much safer to deliver.

Phase 1

  • Source inventory and owner map.
  • Data access decisions.
  • Tenant/RBAC model.
  • Sample data fixtures and known cases.

Phase 2

  • Connector build.
  • Document ingestion and RAG index.
  • Context Builder and audit.
  • Evaluation cases.

Phase 3

  • Agent MVPs.
  • Human review gates.
  • Operational dashboards.
  • Pilot readiness review.
