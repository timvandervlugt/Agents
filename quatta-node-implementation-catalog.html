Tile-by-tile implementation

What every flowchart node means and how to build it

This catalog turns every important tile in the Quatta architecture map into an implementation guide. It is intentionally practical: what to install, what to configure at a customer, what data contract to preserve, and how to prove it is production-ready.

Security items such as tenant filters, RBAC, document versions and citations are controls applied across the retrieval and agent flows. They are not downstream dataflow steps. That is why the flowcharts show them as a control band rather than arrows from individual processing nodes.

Customer Systems: ERP, WMS, SQL, Excel, SharePoint, APIs, Local Drops and Documents

Customer SystemsRead-only by defaultSystem of record remains external

What It Is

The operational source landscape Quatta reads from. Quatta does not replace these systems and does not make them secondary databases.

What To Implement

  • Inventory source inventory per tenant.
  • Read-only connection method per system.
  • Governed upload/SFTP/local-drop path for files.
  • Source owner, SLA, freshness and permission model.
  • Sample fixtures for each source type.

Done Criteria

  • Every source has an owner and access method.
  • No write credentials are used by default.
  • Queries return source IDs and timestamps.
  • Unavailable sources are visible in responses.

Auth / RBAC / Tenant and Role Filters

Microsoft Entra IDFastAPI middlewarePostgreSQL role map

What It Is

Authentication proves who the user is. RBAC and tenant filters decide which sites, customers, articles, orders, claims and documents they can use.

What To Implement

  • Register an Entra application.
  • Validate OIDC tokens in FastAPI.
  • Map Entra groups to Quatta roles.
  • Store tenant, site, customer and document ACL rules.
  • Apply filters before SQL, API and vector retrieval.

Controls

  • Denied permission checks are audited.
  • Vector retrieval never runs without tenant and ACL metadata filters.
  • Document ACL changes trigger hide or re-index behavior.

FastAPIVertex AI GeminiOpenAI optionalAzure OpenAI optional

What It Is

The single control point for model calls, prompt policy, tool permissions, response schemas and safety checks.

What To Implement

  • Provider interface for Gemini, OpenAI and Azure OpenAI.
  • Prompt templates with version IDs.
  • Tool allowlists per workflow.
  • Response validation with citations and confidence.

Monitoring

  • Safety outcome and validation errors.
  • Prompt version and policy version.

Structured Retrieval / Semantic Map

SQL/API retrievalCanonical schemaQuery allowlist

What It Is

Live retrieval of operational facts such as stock, movements, orders, deliveries, tickets and claim status.

What To Implement

  • Map source fields to canonical entities.
  • Define allowlisted query templates.
  • Parameterize item, site, customer, order and date filters.
  • Return row counts, source timestamps and provenance.

When To Use

  • Use structured retrieval for exact facts and calculations.
  • Use RAG only for policy, contract and narrative evidence.

Document AI / OCR Extraction

Vertex AI Document AICloud Run workerDocument metadata

What It Is

OCR and layout extraction for PDFs, scans, contracts, PODs, SOPs, photos and claims evidence before indexing.

What To Implement

  • Stage raw files in tenant-scoped Google Cloud Storage before OCR.
  • Create the right Document AI processor per document class.
  • Store extracted text, tables, page numbers and confidence.
  • Preserve document ID, version, modified timestamp, storage object ID and ACL.
  • Send failed or low-confidence OCR to review.

Acceptance Criteria

  • Every extracted passage points to page and document version.
  • Deleted or permission-revoked documents stop being retrievable.
  • OCR failures are visible in ingestion metrics.

Chunking

Document ingestionSemantic sectionsCitation quality

What It Is

Splitting extracted documents into retrieval units that preserve meaning, section title, page range and permission metadata.

What To Implement

  • Chunk by headings, clauses, tables and page ranges.
  • Store parent document, section, page range and version.
  • Keep enough overlap for context, but avoid huge chunks.
  • Flag tables and contract clauses explicitly.

Controls

  • Chunking does not grant permissions.
  • ACL and tenant metadata travel with every chunk.
  • Re-chunk when document version changes.

Embeddings / Vertex AI

Vertex AI embeddingsCloud Run ingestionpgvector input

What It Is

An embedding is a vector representation of a chunk, used to find semantically similar SOP, contract and evidence passages.

What To Implement At A Customer

  • Choose and version the embedding model.
  • Embed only permission-scoped, approved document chunks.
  • Store vector plus tenant, ACL, document, chunk and version metadata.
  • Re-embed changed chunks and retire old versions.

Operations

  • Monitor embedding job failures and backlog.
  • Track model version per vector row.
  • Test retrieval against known SOP and contract questions.

pgvector

PostgreSQLVector indexMetadata filters first

What It Is

The vector store inside PostgreSQL. It keeps semantic search close to audit, tenant metadata and operational schemas.

What To Implement

  • Enable the vector extension.
  • Create document_chunks and document_embeddings tables.
  • Index tenant, ACL, document version and source timestamp.
  • Apply metadata filters before similarity ranking.

Done Criteria

  • Cross-tenant retrieval tests return zero rows.
  • Revoked documents disappear from search.
  • Top-k results include scores and citations.

Context Builder

FastAPI serviceSQL/API factsRAG passagesCitations

What It Is

The merge layer that turns structured facts and document passages into a compact, grounded context package for agents.

What To Implement

  • Merge facts, passages, citations, warnings and freshness metadata.
  • Deduplicate evidence and flag contradictions.
  • Limit prompt size without dropping required citations.
  • Return a stable JSON schema to LangGraph.

Acceptance Criteria

  • Every answer can be reconstructed from the context package.
  • Missing, stale or conflicting evidence is visible.
  • Context contains no unauthorized source records.

LangGraph Agent Workflows

LangGraphDeterministic toolsHuman gates

What It Is

Workflow orchestration for multi-step reasoning, tool calls, retries, branching and human checkpoints.

What To Implement

  • State schema per workflow.
  • Approved tool nodes for retrieval and calculations.
  • Model nodes only behind the AI Gateway.
  • Human review nodes for high-risk actions.

Controls

  • Agents do not bypass connectors.
  • Financial and liability calculations are deterministic tools.
  • Every node execution is audited.

Audit Log / OpenTelemetry / Governance

PostgreSQL auditOpenTelemetryCloud Monitoring

What It Is

The ability to reconstruct who asked what, which data was retrieved, which model/tool was used, and why the response was allowed.

What To Capture

  • Trace ID, tenant, user, role and request.
  • Sources queried, row counts, document/chunk IDs and timestamps.
  • Model, prompt version, token/cost estimate and safety result.
  • Permission denials, validation failures and human approvals.

Failure Mode

  • If audit write fails for high-risk flows, fail closed.
  • For low-risk read-only flows, degrade only with explicit warning and retry queue.

Inventory Evidence Retrieval

ERP transactionsWMS movementsTicketsSOP/contract retrieval

What It Is

The data collection phase for the question: why does article X have stock differences?

What To Implement

  • Fetch ERP receipts, shipments, adjustments and reversals.
  • Fetch WMS picks, moves, counts and location changes.
  • Fetch tickets, claims and operational notes.
  • Retrieve SOP and contract clauses for tolerances and rules.

Done Criteria

  • Evidence is aligned by item, site, unit and time zone.
  • Missing source systems are explicit.
  • All evidence has source IDs and timestamps.

Inventory Pattern Analysis, Financial Impact and Root Cause

Python deterministic toolsLangGraphGemini summary

What It Is

Deterministic analysis finds variance patterns and exposure. The model explains the result in business language with citations.

What To Implement

  • Reconcile ERP and WMS event timelines.
  • Detect common variance signals: delayed receipt, unit conversion, manual adjustment, misplaced stock, count timing.
  • Calculate financial exposure with controlled formulas.
  • Generate root cause summary and recommended actions.

Controls

  • Low confidence escalates to human review.
  • AI does not invent missing transactions.
  • Recommendations are no-writeback unless approved separately.

Claims Workflow: Eligibility, Liability and Evidence

Contract rulesDeterministic liabilityEvidence packDraft email

What It Is

The workflow that decides whether a claim is likely eligible, calculates liability and drafts a claim package.

What To Implement

  • Retrieve ticket, delivery, order and proof data.
  • Retrieve contract terms, caps and SLA clauses.
  • Run deterministic eligibility and liability tools.
  • Collect evidence and draft an email with attachments and proof gaps.

Acceptance Criteria

  • AI cannot approve or send claims by itself.
  • Liability formula and contract clause are visible.
  • Evidence package is complete or explicitly marked incomplete.

Human Review

Approval UIAudit trailVersioned draft

What It Is

The approval gate for claim sending, operational corrections and high-risk recommendations.

What To Implement

  • Review screen with evidence, rules, liability and draft text.
  • Approve, edit, reject and request-more-info actions.
  • Immutable final approved version.

Controls

  • No outbound claim without approval.
  • Reviewer identity and timestamp are audited.
  • Edits are stored as final human-authored content.

Approved Send / Export

Email/API connectorOutbound controlStatus tracking

What It Is

The controlled outbound step after human approval, either by email, ticket update, API handoff or export.

What To Implement

  • Approved payload export or email send.
  • Status tracking with external ID.
  • Retry and failure handling.
  • Audit of final payload and destination.

Acceptance Criteria

  • Only approved versions are sent.
  • Failures are visible and recoverable.
  • Final payload can be reconstructed from audit.
